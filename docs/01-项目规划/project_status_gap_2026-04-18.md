# 项目当前进度与目标差距评估

日期：2026-04-18

本文档基于当前仓库实际代码、已归档文档、已完成联调记录整理，不是理想状态规划稿，而是“现在做到哪一步”的工程判断。

---

## 一句话结论

当前项目已经完成了“内容平台一期 + 识别配置后台前两批 + 识别消费层首版 + 安卓端 landmarks 直传联调 PoC”。

如果以“能答辩展示一个可运行系统”为标准，当前进度大约在 **60% - 70%**。  
如果以“达到最初目标里的完整智能手语翻译平台”来衡量，当前真正完成度更接近 **50% - 60%**。

原因很直接：

- 平台层、管理层、配置层已经基本成形
- 但“智能手语翻译”最核心的识别引擎能力，当前仍处在首版 PoC 阶段，还没有到可稳定交付的产品形态

---

## 当前已完成部分

## 1. 平台基础能力

这一部分完成度最高，已经具备稳定基础：

- 用户登录
- 管理员登录
- JWT 鉴权
- 文件上传
- 公告管理
- 评论管理
- 分类管理
- 仪表盘基础统计
- 社区帖子发布、列表、详情、我的发布、审核
- 旧 `lostfound` 业务语义迁移为 `post`

结论：

- 一期“内容平台”已经成型
- 这部分已经能支撑答辩演示中的社区、发布、审核、后台管理

## 2. 管理端识别配置后台

二期前两批已经落地并通过管理端联调验收：

- 基础手势库
- 短语模板
- 全局识别配置
- 动作流管理
- 节点配置
- 输出配置

结论：

- “把 blind 的硬编码配置迁到后台管理”这条主线已经走通了一半以上
- 当前已经不是写死在代码里的纯原型，而是有了后台可维护的数据配置入口

## 3. 安卓端识别消费层联调

安卓端和后端的首轮联调已经打通：

- `bootstrap`
- `session/start`
- `predict`
- `session/reset`
- `session/close`
- `recognition-record/my`

并且 landmarks 直传补充口径已经明确：

- `NORMALIZED_XYZ_0_TO_1`
- `TOP_LEFT`
- `MEDIAPIPE_HANDS_21`
- `SINGLE_HAND_FIRST_ONLY`

结论：

- 识别链路已经从“纯设计”进入“真实联调”
- 安卓端目前可以跑通 `gestureCode` 和 `landmarks` 两种模式

## 4. 识别规则消费层首版

后端当前已经具备首版规则消费能力：

- 支持 session
- 支持基础手势命中
- 支持连续命中计数 `requiredHits`
- 支持简单动作流推进
- 支持输出文本、短语、控制动作的基础返回结构
- 支持识别记录落库

结论：

- 当前已经不是“只有配置，没有执行”
- 但执行层仍然偏首版实现，离最终目标还有明显差距

---

## 距离项目目标的主要差距

## 1. 学习模块基本还没真正开始

项目原目标里，学习模块是重要组成部分，但当前仓库里还没有真正落地：

- 学习首页/客户端学习内容消费未完成
- `learning_article` 相关后端模块未完成
- 学习分类、学习统计、学习记录未完成
- 管理端“学习管理”菜单目标尚未实现

影响：

- 当前系统更像“社区 + 识别配置 + 识别 PoC”
- 还不像“社区 + 学习 + 翻译”的完整双创项目

## 2. 识别引擎仍然是 PoC，不是完整产品级实现

这是当前最大的差距。

目前已支持的基础 landmarks 检测手势只有：

- `is_thumbs_up`
- `is_v_sign`
- `is_four_sign`
- `is_fist`
- `is_ok_sign`

而目标里期待的是：

- 更完整的基础手势库
- 更稳定的基础手势识别
- 单动作与树状动作流的可配置执行
- 更贴近真实使用场景的句子/短语输出

当前还缺：

- handedness / mirrored 真正参与判定
- 多手输入协议
- 更完整的基础手势检测集合
- 更稳定的时序策略
- 更细的置信度使用
- 更完善的控制动作链路
- 更完整的 phrase / output 策略

影响：

- 当前能演示“规则驱动识别”
- 但还不能宣称“智能手语翻译能力已经成熟”

## 3. 统计模块还只是一期口径

当前统计更偏内容平台，不是最终项目口径。

已完成：

- 用户总数
- 帖子总数
- 待审核帖子数
- 评论总数
- 分类数
- 公告数
- 最近 7 天内容趋势

目标里还缺：

- 学习条目数
- 手势词条数
- 近 7 天识别使用趋势
- 识别记录相关分析
- 动作命中排行
- 配置使用效果分析

影响：

- 当前后台仪表盘可用
- 但离“识别产品管理后台”还差一层核心数据闭环

## 4. 社区平台仍缺一些增强功能

当前社区主链路已通，但目标里提到的增强能力尚未完成：

- 收藏
- 学习记录
- 设置
- 一键把识别结果转成帖子内容
- 点赞互动闭环
- 更细的用户资料管理

影响：

- 现在社区是“能用”
- 但还不是“完整产品体验”

## 5. 后端仍有少量工程级收尾问题

目前代码已经可编译、可打包、可联调，但还有一些需要尽快补的工程问题：

- 包结构中仍存在一批历史遗留的目录命名不规范问题
- 部分中文返回消息存在编码痕迹
- `AdminUserController` 当前没有显式调用 `adminTokenUtils.checkAdminLogin()`，这是一个真实的权限风险点
- 旧 `lostfound` 代码虽然默认不对外暴露，但仓库里仍保留大量历史代码，后续维护时容易混淆
- SQL 初始化脚本和增量迁移脚本体系还不够规范，目前更像“全量初始化”而不是“版本化 migration”

## 当前最关键的三个短板

如果只看“对最终项目成败影响最大”的问题，当前最关键的是这三个：

1. 识别引擎能力还不够强
2. 学习模块基本未落地
3. 统计与产品闭环还没形成

也就是说：

- 现在平台壳子已经很好了
- 但“智能手语翻译工具”的核心说服力还需要再往前推进一大段

---

## 接下来最优先的推进顺序

建议按这个顺序继续推进，收益最高：

1. 先补识别引擎实质能力
   - handedness / mirrored
   - 多手协议
   - 扩充基础手势
   - 强化动作流判定稳定性
2. 再补学习模块最小闭环
   - `learning_article`
   - 管理端学习管理
   - 客户端学习页
3. 再补统计闭环
   - 识别趋势
   - 手势命中统计
   - 学习条目统计
4. 最后做产品增强
   - 识别结果转帖子
   - 收藏 / 学习记录 / 设置
   - 更完整的个人中心

---

## 当前建议

如果目标是近期做一次“可交差、可答辩、可演示”的版本，建议你把后续重点放在这两条：

- 先把识别链路从 PoC 提升到“稳定可展示”
- 同时补一个最小学习模块，把系统从“社区 + 识别配置”提升成“社区 + 学习 + 识别”

这样项目整体说服力会明显上一个台阶。
