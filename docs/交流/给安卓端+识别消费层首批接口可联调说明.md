# 给安卓端 + 识别消费层首批接口可联调说明

日期：2026-04-16

提出方：后端主线  
对接方：安卓端 uni-app 客户端

## 本轮结论

安卓端当前需要的识别消费层首批接口，后端已经补齐到可联调状态：

- `GET /api/recognition/bootstrap`
- `POST /api/recognition/session/start`
- `POST /api/recognition/predict`
- `POST /api/recognition/session/reset`
- `POST /api/recognition/session/close`
- `GET /api/recognition-record/my`

## 当前可直接使用的能力

### 1. bootstrap

接口：

- `GET /api/recognition/bootstrap`

当前返回：

- `activeConfig`
- `gestureList`
- `phraseList`
- `flowList`

用途：

- 安卓端进入识别页前一次性拉取启动配置
- 减少首页初始化时多接口拼装

### 2. session/start

接口：

- `POST /api/recognition/session/start`

请求体：

```json
{
  "clientType": "UNI_APP_ANDROID",
  "sceneCode": "HOME",
  "engineType": "MEDIAPIPE_JS",
  "appVersion": "0.1.0"
}
```

返回：

- `sessionId`
- `startedAt`
- `activeConfigId`

说明：

- 当前支持匿名启动
- 如果请求头里带用户 token，后端会尽量把用户 ID 关联到会话与记录

### 3. predict

接口：

- `POST /api/recognition/predict`

当前支持两种调用方式：

#### 方式 A：直接传 `gestureCode`

适用场景：

- 安卓端桥接 `blind` 时，先把 H5/bridge 已识别出的手势直接转发给 Spring Boot
- 用于前后端联调、排查状态流、先打通发布承接链路

请求示例：

```json
{
  "sessionId": "rec_xxx",
  "gestureCode": "is_v_sign",
  "source": "android-bridge"
}
```

#### 方式 B：直接传 `landmarks`

适用场景：

- 安卓端后续准备直接切 Spring Boot 识别消费层

请求示例：

```json
{
  "sessionId": "rec_xxx",
  "frameNo": 12,
  "capturedAt": 1713273600000,
  "source": "mediapipe-js",
  "landmarks": [
    { "x": 0.1, "y": 0.2, "z": 0.0 }
  ]
}
```

说明：

- 当前 landmarks 版先支持单手 21 点
- 当前后端内置了基础手势判定的首批能力，优先覆盖：
  - `is_thumbs_up`
  - `is_v_sign`
  - `is_four_sign`
  - `is_fist`
  - `is_ok_sign`

### 4. predict 返回字段口径

当前返回已兼容安卓端文档里要求的旧字段和新字段：

- `gesture`
- `code`
- `label`
- `locked`
- `state`
- `outputType`
- `displayText`
- `ttsText`
- `controlAction`
- `matchedFlowCode`
- `matchedNodeCode`
- `traceId`

说明：

- 为了兼容当前安卓 bridge 阶段，`gesture / code / label / locked / state` 保留
- 为了兼容后续正式消费层，新增了输出类字段

### 5. session/reset

接口：

- `POST /api/recognition/session/reset`

用途：

- 安卓端点“重置识别”时清空当前动作流上下文与稳定帧状态

### 6. session/close

接口：

- `POST /api/recognition/session/close`

用途：

- 安卓端离开识别页时主动关闭会话

### 7. recognition-record/my

接口：

- `GET /api/recognition-record/my?pageNum=1&pageSize=10`

说明：

- 该接口需要用户登录 token
- 当前仅返回当前用户的识别记录分页

## 当前已完成的后端验证

本轮已完成以下实际验证：

- `bootstrap` 返回成功
- `session/start` 返回成功
- `predict` 在 `gestureCode` 模式下可联调
- `session/reset` 返回成功
- 预测结果可写入 `recognition_record`

已验证场景：

- `is_four_sign` 连续命中后进入锁定态
- `is_v_sign` 可命中单动作示例流 `single_hello`

## 当前联调时一定要注意的两点

### 1. 稳定帧命中次数

当前默认生效配置里：

- `requiredHits = 3`

这意味着无论是 `gestureCode` 模式还是 `landmarks` 模式，同一个手势默认都需要连续命中 3 次才会正式触发。

### 2. 动作流节点最大间隔

当前默认示例动作流里，节点间最大间隔仍然会生效。

例如：

- `flow_thanks`
  - 第一步：`is_four_sign`
  - 第二步：`is_v_sign`

其中第二步需要在允许间隔内继续输入，否则会自动回到 `idle`。

当前示例数据建议安卓端联调时：

- 连续帧间隔尽量控制在 500ms 左右
- 不要超过默认 `maxIntervalMs = 1500`

## 现阶段安卓端建议接法

建议安卓端分两步接：

1. 第一阶段先用 `gestureCode` 模式联调
2. 第二阶段再逐步切换到 `landmarks` 直传

这样可以先把：

- 会话管理
- 识别状态展示
- 发布页识别结果承接
- 个人页识别记录

这些业务链路先跑通。

## 当前已知边界

- 当前 Spring Boot 侧基础手势识别只覆盖首批常用手势，不是完整替代 `blind` 的最终版
- 动作流匹配已支持单动作与基础顺序流，但还不是完整生产级规则引擎
- `recognition-record/my` 依赖用户 token；匿名会话不会出现在“我的记录”里

## 需要安卓端配合的约定

- `predict` 优先保持 `gestureCode` 值与后端库中的 `detectionKey / gestureCode` 一致
- 如果先走 bridge 转发，建议继续用：
  - `is_v_sign`
  - `is_four_sign`
  - `is_fist`
  - 这类基础 key
- 如果要开始切 `landmarks` 直传，请默认按单手 21 点格式发

## 下一步建议

安卓端建议优先联调顺序：

1. `bootstrap`
2. `session/start`
3. `predict(gestureCode 模式)`
4. `session/reset`
5. `session/close`
6. `recognition-record/my`
