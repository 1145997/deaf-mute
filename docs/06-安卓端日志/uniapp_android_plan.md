## 原型能力抽取

从 `blind` 的 `index.html` 可以明确抽取出当前识别 Demo 的核心能力：

- 摄像头实时预览
- MediaPipe 手部关键点采集
- `/predict` 识别结果展示
- 句子缓冲与去重输入
- 删除上一个词、清空句子、重置识别状态
- 无新输入后自动 TTS 播报

同时也要明确它当前的边界：

- 页面仍是桌面浏览器 Demo，不是移动端产品界面
- 布局以“视频 + 信息面板 + 工具按钮”为主
- 交互控件偏调试向，不适合直接用于答辩演示版 App
- MediaPipe 运行方式是浏览器 JS 方案，不能直接视为 uni-app App 端可无改造复用

因此安卓端开发的目标不是“照搬 `index.html`”，而是“保留识别核心能力，重构为移动端首页壳子”。

---

## 安卓端目标

安卓 uni-app 客户端第一阶段目标如下：

- 建立符合答辩展示的移动端整体视觉框架
- 以首页承载识别主流程，整体风格接近参考效果图
- 将 `blind` 的识别状态、句子缓冲、TTS 逻辑迁移为移动端可维护状态流
- 对接现有一期社区接口，为“发布”页预留真实业务入口
- 对接现有二期识别配置结果，为后续识别联调预留可扩展接口层

## 页面规划

### 1. 底部导航结构

安卓端采用 4 Tab 结构：

- 首页
- 学习
- 发布
- 个人

其中“发布”保留中间主操作位，整体布局参考效果图。

### 2. 首页

首页为安卓端第一优先级页面，负责承载核心识别体验。

计划模块：

- 顶部状态栏占位与城市位置胶囊
- 中部圆形识别预览区
- “对准手语动作”引导文案
- 识别结果卡片
- 结果播报按钮
- 识别文本滚动区或最近识别记录区
- 底部 TabBar

首页状态流需覆盖：

- 等待识别
- 识别中
- 命中结果
- 句子缓冲中
- TTS 播报完成
- 识别异常或权限异常

### 3. 学习页

学习页作为第二优先级页面，优先做“可演示、可扩展”的轻量版本。

首版计划：

- 基础手势卡片
- 常用短语分类
- 简单学习指引
- 后续预留学习文章列表区域

说明：

- 当前后端“学习模块”尚未正式编码
- 首版学习页允许采用静态内容 + 已有识别配置数据混合渲染

### 4. 发布页

发布页对接现有一期 `post` 主线能力，承担“识别结果转内容发布”的业务闭环。

首版计划：

- 发布表单
- 分类选择
- 文本内容输入
- 识别结果一键带入
- 图片上传预留

### 5. 个人页

个人页首版以基础能力和后续扩展入口为主。

首版计划：

- 登录状态展示
- 个人信息卡片
- 我的发布入口
- 识别历史占位
- 播报、定位、权限设置入口

## 技术路线

### 1. 工程方案

安卓端采用 uni-app 新建独立工程，建议目录命名保持语义清晰，例如：

- `mobile/uniapp-deafmute`

模板选择结论：

- 基础工程模板选择 `uni-app Vue3/Vite + TypeScript` 官方 CLI 模板
- 状态管理使用 `Pinia`
- UI 组件不直接采用整套 `uni ui项目模板` 起步，而是在 CLI 工程中按需引入 `uni-ui` 组件

选择原因：

- 当前仓库前端主线已采用 `Vue3 + TypeScript + Pinia`
- 本机已具备 `Node 22 / npm 10 / pnpm 10`，适合继续走代码仓库友好的 CLI 路线
- 安卓端后续还需要自定义识别桥接层、状态流与业务页面，CLI 工程更利于版本管理和持续迭代
- `uni ui项目模板` 更适合 HBuilderX 日常快速起项目，但当前项目更需要与现有仓库技术口径保持一致

工程内建议优先建立：

- 页面路由
- 全局主题变量
- 请求封装
- 全局状态管理
- Android 权限申请封装

### 2. 识别接入方案

`blind` 当前使用的是浏览器侧 MediaPipe JS 方案，这里需要单独做桥接验证。

计划分两步推进：

1. 先完成首页 UI、状态流、假数据驱动和接口层抽象
2. 再做识别桥接验证，优先评估以下方案：
   - `web-view` 承载 MediaPipe 页面并与 uni-app 通信
   - H5 页面采集 landmarks，uni-app 页面只负责业务壳子与状态展示
   - 若 App 端性能或权限受限，再评估原生插件桥接方案

当前默认选择：

- 先走 `web-view / H5 bridge` 验证链路

原因：

- 与 `blind` 原型差距最小
- 能最快复用现有 MediaPipe JS 识别方式
- 适合先跑出演示链路，再做性能优化

桥接验证结论：

- 第一轮桥接目标不是直接替换为 Spring Boot 识别接口
- 第一轮应由 uni-app 页面承载移动端壳子，`web-view` 内运行 `blind` 识别页
- `web-view` 将识别结果以消息形式同步给 uni-app 页面
- uni-app 页面负责展示态、句子缓冲、业务跳转和发布承接

这样可以把桥接拆成两层：

- 识别层：`blind` 的 MediaPipe + FastAPI 原型
- 业务层：uni-app App 页面与社区业务

### 3. 接口层规划

安卓端接口层分三组：

- 一期社区接口
  - 登录
  - 分类
  - 帖子发布
  - 我的发布
- 二期识别配置消费接口
  - 当前生效识别配置
  - 基础手势列表
  - 短语模板列表
- 后续识别联调接口
  - 识别会话启动
  - landmarks 提交识别
  - 重置识别状态

首版允许对尚未正式落地的识别消费接口使用 mock 层，避免前端被接口进度阻塞。

当前后端代码判断：

- Spring Boot 主后端当前已具备识别配置查询能力
- 但尚未提供真正的识别消费层接口
- 因此第一轮安卓端桥接不能直接依赖主后端完成 `/predict`

当前可直接消费的数据接口为：

- `GET /api/gesture-library/enabled`
- `GET /api/phrase-template/list`
- `GET /api/recognition-config/active`

动作流当前仍以管理端维护为主，消费层接口待后续补齐。

## 风险与依赖

### 风险点

- 当前仓库还没有现成的 uni-app 客户端工程
- `blind` 的浏览器 MediaPipe 方案在 App 端存在兼容和性能风险
- 学习模块后端未正式落地，学习页首版需要降级方案
- 识别消费层接口还未正式成型，前端需要保留 mock 与适配层

### 依赖项

- 后端后续提供识别消费接口
- 一期 `post` 相关公开接口保持稳定
- 若 `web-view` 方案受限，需要尽早决定是否接入原生插件
